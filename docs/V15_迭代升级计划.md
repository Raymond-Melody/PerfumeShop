### 1.2 核心痛点与风险评估

| # | 问题 | 影响范围 | 风险等级 | 具体表现 |
|---|------|----------|----------|----------|
| **1** | **SQLOLEDB 驱动已弃用** | 全局 | 🔴 高 | 微软自2017年起不再维护SQLOLEDB。未来Windows更新可能移除该Provider，导致全站瘫痪。`connection.asp` 第25行 `Provider=SQLOLEDB` |
| **2** | **SQL注入风险广泛存在** | 全局 | 🔴 高 | 搜索结果显示至少80+处使用 `conn.Execute()` 直接拼接字符串。虽然 `ExecuteParameterizedQuery` 已在V14.6引入（connection.asp L773-822），但实际使用率接近0% |
| **3** | **密码安全不足** | 用户系统 | 🔴 高 | `password_utils.asp` 的自定义 `InternalHash` 非标准算法，等价于自定义HMAC而非bcrypt。`HashPasswordV2` 10轮迭代仅基于256模运算，破解难度远低于标准bcrypt |
| **4** | **API响应格式不统一** | API层 | 🟡 中 | 14个API端点各有独立JSON构建逻辑。`risk_check.asp` 用Dictionary+手动拼接，`get_areas.asp` 用字符串拼接，`track.asp` 用1x1像素图片——缺乏统一约定 |
| **5** | **Application缓存无自动清理** | 全局 | 🟡 中 | `CacheGet`/`CacheSet` 有TTL机制但仅在读取时检查过期。大量过期缓存项持续占用Application内存，无主动回收机制 |
| **6** | **日志系统碎片化** | 全局 | 🟡 中 | `error_handler.asp` 写入文件日志，`tracking_utils.asp` 写入SiteSettings表，`debug_checkout.log` 独立存在——三种日志路径互不关联 |
| **7** | **无自动化测试** | 全局 | 🟡 中 | 回归测试完全靠手工。200+文件的任何修改都可能引入未知回归 |
| **8** | **PWA资源不完整** | 前端 | 🟠 低 | `manifest.json` 引用的8个图标文件均不存在；`sw.js` 缓存策略未区分静态资源与API请求 |

### 1.3 现有架构优势（应保留）

| 优势 | 说明 |
|------|------|
| **模块化成本引擎** | `cost_engine.asp` 实现三级成本自动传导（原材料→香调→产品→订单），Scripting.Dictionary内存缓存设计优秀，预加载后零DB查询 |
| **会员推荐制** | V14.6的 `member_utils.asp` 推荐系统设计完善：SHA-256签名Token、使用次数限制、祖先链条追溯、每日生成限制 |
| **配置集中化** | `config.asp` 统一管理版本号、密钥、常量、CDN、CSP等所有全局配置 |
| **安全防护基础** | CSRF Token、CSP Header、HSTS、速率限制、Cookie签名等多层防护已到位 |
| **响应式设计** | 完备的移动端适配，骨架屏加载、懒加载、暗黑主题等现代UX模式 |

## 三、功能开发规划

### 3.1 P0 — 安全与基础设施（必须完成，第1-2周）

#### P0-1：数据库驱动升级（预估2天）

**文件**: `includes/connection.asp` L25
**变更**: `Provider=SQLOLEDB` → `Provider=MSOLEDBSQL`

```
当前: conn.Open "Provider=SQLOLEDB;Server=localhost\YOURPERFUME;Database=PerfumeShop;Integrated Security=SSPI;"
目标: conn.Open "Provider=MSOLEDBSQL;Server=localhost\YOURPERFUME;Database=PerfumeShop;Integrated Security=SSPI;"
```

**实施步骤**:
1. 下载安装 MSOLEDBSQL 驱动 (Microsoft OLE DB Driver for SQL Server 18.x)
2. 在 `connection.asp` 增加双Provider配置常量，支持一键切换回滚
3. 全站冒烟测试：首页/产品列表/产品详情/购物车/结算/登录/注册
4. 后台模块抽检：operation/orders, inventory/index, finance/reports, techcenter/product_settings

**回滚方案**: 配置 `USE_MSOLEDBSQL = False` 即可切回 SQLOLEDB

#### P0-2：密码安全升级（预估1.5天）

**文件**: `includes/password_utils.asp`

**问题**: 当前 `InternalHash` 使用 `(char*7 + prev) Mod 256` 循环，本质是自定义CRC变种，非密码学哈希。

**方案**: 引入标准bcrypt兼容的ASP实现或使用SQL Server内置 `HASHBYTES('SHA2_512', ...)` 配合盐值

```
' V15 密码哈希升级
Function HashPasswordV3(password)
    ' 使用SQL Server内置SHA-512 + 多轮迭代 + Salt + Pepper
    ' 格式: V3$<salt>$<hash>
End Function
```

**兼容策略**: V2哈希保留，用户登录时自动升级到V3（静默迁移）

#### P0-3：统一数据访问层（预估5天）

**新建文件**:
- `includes/dal.asp` — 核心DAL（参数化查询工厂）
- `includes/dal_products.asp` — 产品相关数据操作
- `includes/dal_orders.asp` — 订单相关数据操作
- `includes/dal_users.asp` — 用户相关数据操作

**核心接口设计**:

```vbscript
' dal.asp 核心函数

' 查询单条记录（返回Dictionary）
Function DAL_GetRow(sql, paramArray)
    ' 自动使用 ExecuteParameterizedQuery
End Function

' 查询列表（返回Recordset，兼容现有代码）
Function DAL_GetList(sql, paramArray)
End Function

' 执行非查询（INSERT/UPDATE/DELETE，返回受影响行数）
Function DAL_Execute(sql, paramArray)
End Function

' 获取标量值
Function DAL_GetScalar(sql, paramArray)
End Function

' 检查记录是否存在
Function DAL_Exists(tableName, whereClause, paramArray)
End Function
```

**迁移策略**: 渐进式替换，优先迁移高频访问路径：
1. 用户登录/注册 → `dal_users.asp`
2. 产品浏览/搜索 → `dal_products.asp`
3. 购物车/结算 → `dal_orders.asp`
4. 后台订单管理 → `dal_orders.asp`

**预期效果**: V15结束时，核心路径SQL注入风险归零

### 3.2 P1 — 架构优化（第3-4周）

#### P1-4：结构化日志系统（预估2天）

**新建**: `includes/logger.asp`

```
核心功能:
- LOG_DEBUG(msg)  — 调试信息
- LOG_INFO(msg)   — 常规操作
- LOG_WARN(msg)   — 警告事件
```

**迁移策略**: 替换 `error_handler.asp` 的 `EH_WriteLog` 调用，逐步替换 `Session("LastDBError")` 模式

#### P1-5：API标准化（预估3天）

**新建**: `includes/api_response.asp`

**统一JSON响应格式**:
```json
{
  "code": 0,           // 0=成功，非0=错误码
  "message": "success",
  "data": { ... },     // 业务数据
  "timestamp": "2026-06-24T14:30:00Z",
  "requestId": "a1b2c3d4"
}
```

**错误码规范**:
| 范围 | 含义 |
|------|------|
| 0 | 成功 |
| 1000-1999 | 认证/授权错误 |
| 2000-2999 | 参数验证错误 |
| 3000-3999 | 业务逻辑错误 |
| 4000-4999 | 数据库错误 |
| 5000-5999 | 文件/上传错误 |

**影响文件**: 14个 `/api/*.asp` 端点全部迁移到统一格式

**兼容策略**: 现有API路径保持不变，新格式通过响应头 `X-API-Version: v1` 标识。前端JS通过尝试解析新格式，失败时回退旧格式

## 四、向后兼容性策略

### 4.1 核心原则

> **不中断现有业务、不修改现有URL、不强制前端更新**

### 4.2 具体措施

| 变更项 | 兼容策略 |
|--------|----------|
| **SQLOLEDB→MSOLEDBSQL** | `connection.asp` 配置开关 `USE_MSOLEDBSQL`，默认false，验证后切换true。两套连接字符串并存 |
| **DAL层引入** | 新建文件不影响现有代码。旧 `ExecuteQuery()` 函数保留并标记 `@deprecated`。新代码使用DAL，旧代码逐步迁移 |
| **API标准化** | 旧API端点不变。新增 `/api/v1/` 前缀新端点。响应新增 `X-API-Version` 头，前端渐进适配 |
| **密码升级** | V2→V3静默迁移。用户登录时自动重新哈希存储。V2验证逻辑保留作回退 |
| **缓存升级** | `CacheGet`/`CacheSet` 函数签名不变，内部委托到 `cache_manager.asp`。旧调用无需修改 |
| **日志升级** | `EH_WriteLog` 保留并委托到新 `logger.asp`。新增 `LOG_*` 函数为新推荐方式 |
| **数据库表变更** | 只新增表/列，不修改现有结构。所有迁移脚本可回滚 |
| **前端JS** | 新API响应格式通过 `X-API-Version` 头检测，前端逐模块适配。旧版浏览器回退兼容 |

### 4.3 功能开关（Feature Flag）

在 `config.asp` 中新增：

```vbscript
' V15 Feature Flags
Const FEATURE_MSOLEDBSQL = False        ' P0: 数据库驱动升级
Const FEATURE_DAL_ENABLED = False       ' P0: DAL层启用
Const FEATURE_API_V1 = False            ' P1: API v1格式
Const FEATURE_SSE_NOTIFICATIONS = False ' P2: 实时通知
Const FEATURE_EMAIL_NOTIFICATIONS = False ' P2: 邮件通知
Const FEATURE_ANALYTICS_DASHBOARD = False ' P2: 数据分析
Const FEATURE_I18N = False              ' P3: 国际化
```

所有新功能默认关闭，经完整测试后逐项开启。

## 六、风险控制措施

### 6.1 风险矩阵

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| MSOLEDBSQL兼容性问题 | 中 | 高 | 双Provider配置开关，5分钟可回滚。测试覆盖所有SQL操作类型 |
| DAL层性能下降 | 低 | 中 | ADODB.Command参数化查询与conn.Execute性能差异<5%。基准测试验证 |
| 密码迁移导致用户无法登录 | 低 | 高 | 保留V2验证路径，V3静默升级。测试覆盖V1/V2/V3全兼容 |
| API变更破坏前端 | 中 | 中 | 响应头版本标识，前端渐进适配，旧格式兼容保留 |
| 日志文件磁盘溢出 | 低 | 中 | 按日轮转 + 30天自动清理 + 1MB大小截断 |
| 实时通知增加服务器负载 | 中 | 低 | 长轮询30秒间隔，连接池限制最大并发50。必要时可关闭FEATURE_SSE |
| 邮件被标记为垃圾邮件 | 中 | 低 | SPF/DKIM配置建议（需DNS配合）。邮件频率限制 |
| 数据分析查询影响主库性能 | 低 | 中 | 预聚合表 + 缓存，避免实时大查询。非高峰时段执行SQL Job |

### 6.2 质量保障

1. **每个里程碑完成后进行冒烟测试**：核心业务流程（注册→登录→浏览→加购→下单→支付）自动化验证
2. **SQL注入扫描**：DAL层覆盖后，对所有用户输入点进行自动化扫描
3. **性能基线对比**：V14.6 → V15 首页加载时间、API响应时间对比
4. **代码审查清单**：每个PR必须通过安全检查（参数化查询、XSS防护、CSRF Token）

### 6.3 回滚预案

```
V15发布前:
  git tag v14.6-final-backup  (V14.6最终快照)
  git branch backup-v15-rollback  (V15回滚分支)

紧急回滚步骤:
  1. git checkout v14.6-final-backup
  2. 部署到IIS
  3. config.asp 中所有 FEATURE_* 设为 False
  4. 数据库: 执行 database/v15_rollback.sql（如有schema变更）

预计回滚时间: <10分钟
```

## 七、V16 远期规划衔接

### 7.1 V15 → V16 承接关系

```
V15 完成的基础            V16 可启动的工作
─────────────            ────────────────
DAL层 + API标准化    →   ASP.NET Core Web API 替代 Classic ASP API层
统一日志 + 监控        →   Serilog + Application Insights
缓存管理器             →   Redis 分布式缓存
API v1               →   Swagger/OpenAPI 自动文档
邮件队列              →   Azure Service Bus / RabbitMQ
数据分析预聚合         →   PowerBI / Metabase 直接连接
```

### 7.2 V16 规划概要（不在V15范围内）

```
V16: 平台现代化 — ASP.NET Core 增量迁移

Phase 1 (V16.0): API层迁移
  - API端点迁移到 ASP.NET Core Web API
  - IIS反向代理，Classic ASP与.NET Core共存
  - Entity Framework Core替代ADODB

Phase 2 (V16.1): 后台管理迁移
  - 后台页面迁移到 React/Vue SPA
  - JWT认证替代Session
  - RBAC权限管理重构

Phase 3 (V16.2): 前台迁移 + 完整迁移
  - 前台迁移到 Razor Pages / Blazor Server
  - Redis缓存替代Application
  - CI/CD流水线 (GitHub Actions / Azure DevOps)

触发条件: V15 稳定运行 4 周 + DAL覆盖率 ≥ 70%
```

### 7.3 V15架构决策对V16的影响

| V15决策 | 对V16的影响 |
|---------|------------|
| DAL层实现为ASP函数 | V16迁移到EF Core时，DAL函数签名可1:1映射到Repository方法 |
| API v1 JSON格式 | V16的ASP.NET Core API使用相同格式，前端零改动 |
| 邮件队列基于数据库表 | V16可平滑升级到消息队列，表结构不变 |
| i18n字典文件 | JSON格式字典文件可在.NET Core中直接复用 |
